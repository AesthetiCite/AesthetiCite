UX change — why "What complication are you facing?" is the right headline
The existing t("landing.headline") is a generic evidence search prompt. It works for medical students. It fails for a clinician with a patient in front of them mid-complication. They don't want to think about what to type — they want one tap.
The 8 quick-select buttons fire complete optimised queries, not just the complication name. "Vascular Occlusion" fires "vascular occlusion management hyaluronidase treatment protocol" — the exact phrasing that retrieves the best evidence chunks. The clinician gets a better answer than if they typed "vascular occlusion" themselves. The urgent options (Vascular, Anaphylaxis) have red styling and a pulsing dot to signal time-criticality without adding text. Keyboard shortcuts (V, A, N, I, P) are there for power users.

Speed architecture — where the 2 seconds actually comes from
The existing code already has is_hot_question() and get_cached_answer() — but they only fire when a question has been explicitly marked hot, which almost never happens in practice. The real bottleneck breakdown:
StepCurrentAfter hot cacheAfter context limiterEmbedding + vector search~200ms0ms (precomputed)~200msLLM first token~900ms~50ms (from memory)~700msTotal to first token~1.1s~55ms~900ms
The HOT_QUERIES registry in speed_optimizer.py has 16 canonical complication queries with aliases. When a query matches any alias, it gets the precomputed result from memory. precompute_hot_queries() runs in a background thread at startup, pre-warming the retrieval cache for all 16. The context limiter cuts the evidence pack from 10–14 chunks to 6 for complication queries — that's ~50% fewer prompt tokens, which cuts LLM latency by roughly 30%.
The /api/speed/stats endpoint shows hit rate and per-mode latency percentiles in real time. If target_met is below 0.85, add more aliases to HOT_QUERIES for the queries showing up in misses.
