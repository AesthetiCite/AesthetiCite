PATCH 2 — server/routes.ts
================================================================================

ADD inside registerRoutes():

    // ── Clinical Reasoning (Glass Health-inspired) ────────────────────────
    app.post("/api/reasoning/stream", async (req, res) => {
      try {
        res.setHeader("Content-Type", "text/event-stream");
        res.setHeader("Cache-Control", "no-cache");
        res.setHeader("Connection", "keep-alive");
        res.setHeader("X-Accel-Buffering", "no");

        const headers: Record<string, string> = { "Content-Type": "application/json" };
        if (req.headers.authorization) {
          headers["Authorization"] = req.headers.authorization as string;
        }
        const fastApiRes = await fetch(`${PYTHON_API_BASE}/api/reasoning/stream`, {
          method: "POST",
          headers,
          body: JSON.stringify(req.body),
        });
        if (!fastApiRes.ok || !fastApiRes.body) {
          res.write(`data: ${JSON.stringify({ type: "error", message: "Reasoning unavailable" })}\n\n`);
          res.end();
          return;
        }
        const reader = (fastApiRes.body as any).getReader();
        const decoder = new TextDecoder();
        while (true) {
          const { done, value } = await reader.read();
          if (done) break;
          res.write(decoder.decode(value, { stream: true }));
        }
        res.end();
      } catch (error) {
        res.write(`data: ${JSON.stringify({ type: "error", message: "Stream failed" })}\n\n`);
        res.end();
      }
    });

    app.post("/api/reasoning",       (req, res) => proxyToFastAPI(req, res, "/api/reasoning"));
    app.get("/api/reasoning/cache",  (req, res) => proxyToFastAPI(req, res, "/api/reasoning/cache"));

================================================================================
PARALLEL FETCH ARCHITECTURE
================================================================================

The key architectural decision: reasoning runs IN PARALLEL with the
workflow, not sequentially. Here's the call pattern in the frontend:

    // In handleSubmit() in complication-decision.tsx:
    // These two fetches fire simultaneously:

    // 1. /api/decide → fast deterministic result (<300ms)
    mutation.mutate(query);  // sets result → renders workflow immediately

    // 2. /api/reasoning/stream → LLM reasoning (~2-4 seconds)
    // Handled automatically by ClinicalReasoningSection autoFetch=true
    // Streams into its own card as the LLM generates

    // Result: User sees workflow in <300ms.
    //         Reasoning section fills in over the next 2-4 seconds.
    //         Zero added latency to the critical path.


================================================================================
HOW THE LLM PROMPT DIFFERS FROM THE BUILD PLAN
================================================================================

Build plan prompt (generic):
  "Return: diagnosis, reasoning[], differentials[], confidence"

What was built (aesthetic-medicine-specific):
  - Includes region, procedure, product, time_since_injection
  - Includes top-3 retrieved evidence chunks as grounding context
  - Returns 8 fields: diagnosis, confidence, confidence_why,
    reasoning (4 labeled steps), key_signs, against_signs,
    differentials (with exclude_reason + likelihood per differential),
    red_flags, limitations, evidence_refs
  - Uses GPT-4o (not mini) — clinical reasoning needs the best model
  - Uses JSON mode (response_format: json_object) — guaranteed valid JSON
  - Temperature 0.15 — consistent, not creative

The prompt instructs the model to:
  - Be specific to aesthetic medicine (not generic medicine)
  - Explain WHY each differential is excluded (not just list them)
  - Be honest in limitations
  - Only use signs/symptoms actually present in the case
  - Reference known complication patterns
